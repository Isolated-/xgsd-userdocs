### Concept

**Templating** lets you use runtime data to build values before a block runs.

---

### 1. Using templates

Templates can be used in `input` and `env`.

Example:

```yaml
data:
  name: World
blocks:
  - run: greet
    input:
      message: "Hello {{ctx.data.name}}"
```

`greet` will receive:

```json
{
  "name": "World",
  "message": "Hello World"
}
```

---

### 2. Available data (context)

Templates can access the following values:

| property          | note                                          |
| ----------------- | --------------------------------------------- |
| `ctx.id`          | unique run identifier                         |
| `ctx.name`        | project name (or `unknown`)                   |
| `ctx.hash`        | config hash                                   |
| `ctx.projectPath` | absolute path to the project folder           |
| `ctx.entry`       | absolute path to the entry file               |
| `ctx.start`       | ISO timestamp when the run started            |
| `ctx.data`        | project input data (including `-d` overrides) |
| `ctx.mode`        | execution mode (`chain` or `async`)           |
| `ctx.concurrency` | concurrency limit                             |
| `block.idx`       | position in the sequence                      |
| `block.run`       | exported function name                        |
| `block.name`      | block name (defaults to `block.run`)          |

Templates can only access the values listed above. Internal runtime state is not available.

---

### 3. Full vs inline templates

If a value is entirely a template, the original type is preserved:

```yaml
input:
  data: "{{ctx.data}}"
```

If a template is part of a string, it is converted to a string:

```yaml
input:
  message: "Hello {{ctx.data.name}}"
```
