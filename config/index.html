Getting started

A minimal valid config looks like:

  • yaml
  • JSON
config.yaml
blocks:
  - run: greet
config.json
{
  "blocks": [
    {
        "run": "greet"
    }
  ]
}

Everything else is optional.

warning

By default config.yaml is assumed to be your project config file. If you're using JSON or have renamed your file, you'll need to use xgsd run -c {file}.

It's recommended to use config.yaml for simplicity.

Templating​

Templates can be used in input and env.

Example:

  • yaml
  • JSON
config.yaml
data: 
  name: World
blocks:
  - run: greet
    input:
      message: Hello {{ctx.data.name}}
config.json
{
  "data": {"name": "World"}
  "blocks": [
    {
      "run": "greet",
      "input": {
        "message": "Hello {{ctx.data.name}}"
      }
    }
  ]
}

greet will receive:

{
  "name": "World",
  "message": "Hello World"
}
info

In v0.5, Helpers were replaced with Templates to remove logic from config and make xGSD more predictable.

Instead of:

${{ctx.data}}

Use:

{{ctx.data}}

Any functions like censor or slice have been removed.

Available data (context)​

Templates can access the following values:

propertynote
ctx.idunique run identifier
ctx.nameproject name (or unknown)
ctx.hashconfig hash
ctx.projectPathabsolute path to the project folder
ctx.entryabsolute path to the entry file
ctx.startISO timestamp when the run started
ctx.dataproject input data (including -d overrides)
ctx.modeexecution mode (chain or async)
ctx.concurrencyconcurrency limit
block.idxposition in the sequence
block.runexported function name
block.nameblock name (defaults to block.run)

Templates can only access the values listed above. Internal runtime state is not available.

Full vs inline templates​

  • yaml

If a value is entirely a template, the original type is preserved:

config.yaml
input:
  data: "{{ctx.data}}" # data is an object
  • yaml

If a template is part of a string, it is converted to a string:

config.yaml
input:
  name: "{{ctx.data.name}}" # name is a string (e.g. "World")

Environment Variables​

Static variables​

Static values can be defined directly in your config.yaml:

  • yaml
  • JSON
config.yaml
# inside your blocks:
env:
  MY_VAR: 1234
config.json
{
  "env": {
    "MY_VAR": 1234
  }
}

This is suitable for non-sensitive configuration.

Resolved variables​

Sensitive values can be resolved at runtime instead of being stored in your config.

  • yaml
  • JSON
config.yaml
# inside your blocks:
env:
  MY_VAR:
    source: env
    ref: SENSITIVE
config.json
{
  "env": {
    "MY_VAR": { "source": "env", "ref": "SENSITIVE" }
  }
}

This reads from process.env.SENSITIVE.

You can also load them from files:

  • yaml
  • JSON
config.yaml
# inside your blocks:
env:
  # dotenv style
  MY_VAR_ENV:
    source: file
    path: myvars.env
    ref: my_var
  # json
  MY_VAR_JSON
    source: file
    path: myvars.json
    ref: my_var
config.json
{
  "env": {
    "MY_VAR_ENV": { "source": "file", "ref": "my_var", "path": "myvars.env" },
    "MY_VAR_JSON": { "source": "env", "ref": "my_var", "path": "myvars.json" }
  }
}

Resolved values are injected into the runtime environment. Read variables like normal with process.env.MY_VAR.

This does not affect projects ran with --dev or --local.
