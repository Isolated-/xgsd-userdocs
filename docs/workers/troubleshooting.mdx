## Debugger

Need to debug inside the process?

Create simple middleware like:

```js
const debug = (ctx, next) => {
  // do something with ctx (before)
  console.log('after', ctx);

  await next()

  console.log('after', ctx)

  if (ctx.error) {
    console.error(ctx.error.message ?? 'unknown error')
  }

  // ...
};

export const middleware = () => [debug]
```

Avoid debugging with lots of middleware registered.

If you're debugging middleware failures then register them in layers to isolate the problem.

## Errors

**Workers** are fault-tolerant by design but unexpected things happen.

### System

See [**WorkerErrorCode**](/api/enumerations/WorkerErrorCode) for what errors are thrown and when.

All system errors result in a rejection (minus `CODE_WORKER_GUARD`) so you should wrap `await transport()` in a try/catch block.

This is intentional as it prevents leaking system failures to users of your apps.

### Usercode

When there's an error in your application, it'll be wrapped:

```json
{
  "version": "v1",
  "ok": false,
  "result": null,
  "error": {
    "original": {},
    "name": "Error",
    "message": "bad things happen",
    "stack": "..."
  },
  "duration": 820.5204840000006
}
```

:::warning
`error.original` may only be available inside your process. Use middleware to intercept and handle it if needed.
:::

These will be untagged and instead make best effort to normalise and serialise the error thrown.

Thrown when user forces process to exit (CTRL+C).

## CODE_INVALID_DATA

Data cannot be safely serialised for transport.

### Causes

- Circular references
- Non-serialisable values (e.g. `BigInt`, functions)
- Structured clone incompatibility

### Fixes

To keep worker functions pure, you can transform any circular or non-serialisable values using a transformer:

```js
const transformer = async (ctx, next) => {
  await next();

  // here's how you'd do it for an Axios-like response
  ctx.result = ctx.result?.data;
};
```

---

## CODE_INVALID_MIDDLEWARE_FUNCTION

Middleware provided is not a valid function.

A middleware function is defined as:

```js
const A = async (ctx, next) => {};
```

Your `middleware()` should look like:

```js
// make sure it's exported
// name must be "middleware"
export const middleware = () => [A, B, C];

// or
export const middleware = async () => [A, B, C];
```

### Causes

- Non-callable middleware
- Incorrect plugin shape
- Exported `middleware()` function is not valid

## CODE_INVALID_ENTRY_FILE

The entry file cannot be resolved, loaded, or parsed.

Since **v1.0.2** — these errors now provide more information and the stack trace so you don't need to trace signals.

### Causes

- File does not exist
- Syntax error in module
- Unsupported module format (CJS/ESM mismatch)

---

## CODE_INVALID_DEFAULT_FUNCTION

The module `default` export is not a function.

A worker function is defined as:

```ts
export default async (data) => {};
```

### Invalid examples

- Object export
- `null` or `undefined`
- Primitive values

## CODE_FATAL_ERROR

If you see this error then you've probably just discovered an edge case Workers.js wasn't prepared for.

Please [**report this problem**](https://github.com/Isolated-/xgsd-workers/issues/new).

### Causes

- Unhandled exceptions/rejections from within the worker process/container.

### Note

This typically won't be anything to do with your code or something you can fix.

## Worker guard

Anything to do with `CODE_WORKER_GUARD` means that one or more of your limits were exceeded and the process has been killed.

If this is unexpected, the default limits are:

```json
{
  "limits": {
    "ttl": 5000,
    "memory": 64
  }
}
```

Worker guard activations don't result in a fatal error (e.g. promise gets rejected), instead the return a normal response. This is designed to allow the transport to determine what to do.

Since **v1.0.3** — you can override this:

```js
const transport = createTransport({
  limits: {
    on: "throw",
  },
});
```

which will result in the promise rejecting like other system errors.

:::info
By default **heap usage** is used when limiting memory.

You can use:

```js
const transport = createTransport({
  entry: "./worker.js",
  limits: {
    memory: {
      limitMB: 64,
      strategy: "rss",
    },
  },
});
```

If you prefer RSS.

:::

## Serialisation/Deserialisation

If you're seeing `{}` in a `result` or `error` that most likely means either:

1. You're returning an instance or other unserialisable type.
2. The value you're returning is too large for serialisation.

A trade off to using child processes is that information isn't passed in like arguments to a function. Instead data is serialised and sent over IPC.

As a result, values _must_ be serialised or they'll be reduced to `{}`.

### Errors

As of **v1-beta** a fatal error is thrown if your data cannot be serialised.

The error code for these errors is `CODE_INVALID_DATA` and this is thrown if `result` or `error` cannot be serialised.

If you see this error you should ensure your worker and middleware functions aren't returning any non-serialisable values.

## ESM/CommonJS

We only test and build against ESM.

If you're using CommonJS consider updating your project to ESM.

You _may_ be able to use CommonJS but it's not recommended.

If you're having issues with ESM [**please report issues here**](https://github.com/Isolated-/xgsd-workers/issues/new).

## TypeScript

Until **v1.0.3** a number of important types were not exported.

This might of made usage hard without resorting to `as any` everywhere.

This has been addressed and will continue to be improved to ensure TypeScript is always correctly supported.
