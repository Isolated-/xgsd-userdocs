Workers

Quick Overview​

Workers are functions that execute in an isolated environment.

They are intentionally limited and have no concept of Projects or Blocks.

Instead of a project-level config.yaml, you create a worker-level worker.json.

Feature@xgsd/runtime@xgsd/workers
Best forFull orchestration systemsServerless / isolated execution
ScopeProject-level systemsSingle execution
ComplexityHigherMinimal
ExtensibilityHigh (plugins/events)Low (intentionally constrained)
Execution modelStateful workflow runtimeStateless worker runtime
Primary strengthCoordination & structureSpeed & isolation

Example worker.json:

worker.json
{
  // not complete
  "entry": "index.ts",
  "bundler": {
    "enabled": true,
    "cache": {
      "strategy": "change"
    }
  },
  "http": {
    "enabled": true
  },
  "limits": {
    "ttl": 1000,
    "memory": 100
  },
  "metadata": {}
}

Example index.ts (or .js):

index.ts
export default async function (data: any) {
  const url =
    data.url ??
    "https://timeapi.io/api/Time/current/zone?timeZone=Europe/London";

  const res = await fetch(url);
  const json = await res.json();

  return json;
}

Goal​

Reduce memory usage and time wasted starting/stopping processes.

After refactoring a lot of xGSD for v0.7, I realised there's a lot that xGSD is doing that's not truly needed.

Workers will focus on need to have vs nice to have.

Schema​

Workers are designed around a static schema:

schema.json
{
  "version": "v1",
  "ok": true,
  "result": {
    // data
  },
  "error": {
    // error
  },
  "duration": 0
}

Transports​

Workers are transport-agnostic and you can create new transport options using createHandler():

yourapp.ts
import { createHandler } from "@xgsd/workers";

const config = getWorkerConfig();
const handler = createHandler(config);

// inside your transport (e.g. Express route):
async function callback(req, res) {
  const data = req.body;

  const result = await handler({
    data,
    env: {
      // env vars
    },
    cwd: "path/to/worker",
  });
}

This will be simplified as development continues.

Errors​

Error handling is a core concern of Workers and is something it'll do very well.

System​

System errors are thrown when something goes wrong before or after your code has finished running.

  • CODE_WORKER_GUARD - this error is thrown whenever your process limits are exceeded (ttl/memory).
  • CODE_INVALID_ENTRY_FILE - thrown when your entry file is missing, or an error prevents load.
  • CODE_INVALID_DEFAULT_FUNCTION - thrown when you don't export a default function, or if your default export is not a function.
  • CODE_INVALID_MIDDLEWARE_FUNCTION - thrown when one or more of your middleware functions are invalid (e.g. not functions).
  • CODE_BUNDLE_ERROR - thrown when bundling errors occur (currently unused).
  • CODE_FATAL_ERROR - thrown when an unhandled exception/rejection is caught (currently unused)

Usercode​

When there's an error in your application, it'll be wrapped:

{
  "version": "v1",
  "ok": false,
  "result": null,
  "error": {
    "original": {},
    "name": "Error",
    "message": "bad things happen",
    "stack": "..."
  },
  "duration": 820.5204840000006
}

Middleware​

Extensions aren't supported in Workers, instead a lightweight middleware system is provided:

// define your middleware
const logger = (ctx: RuntimeContext, next: Next) => {
  console.log("my function just started");

  // optionally mutate before state:
  ctx.input = { myVar: true };

  await next();

  // do something with result:
  ctx.result = { ...ctx.result, addedByMe: true };

  console.log("my function has ended!");
};

// this function is called just before your function runs
// it should always return an array of middleware functions
export const middleware = () => [logger];
