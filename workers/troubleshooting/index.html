Troubleshooting

Remember to set cache to never before you start troubleshooting.

Seriously.

A surprising amount of "xGSD is broken" moments turned out to be:

  • stale bundles
  • stale config
  • stale output
  • or me forgetting cache was enabled (usually this!)

Please report any problems that you find - I'll get them fixed ASAP.

Include a brief summary of your problem and any error stack/trace info available.

Debugger​

Need to debug inside the process?

Create simple middleware like:

const debug = (ctx, next) => {
  // do something with ctx (before)
  console.log('after', ctx);

  await next()

  console.log('after', ctx)

  if (ctx.error) {
    console.error(ctx.error.message ?? 'unknown error')
  }

  // ...
};

export const middleware = () => [debug]

Avoid debugging with lots of middleware registered.

If you're debugging middleware failures then register them in layers to isolate the problem.

Errors​

Workers are fault-tolerant by design but unexpected things happen.

System​

System errors are thrown when something goes wrong before or after your code has finished running.

  • CODE_WORKER_GUARD - this error is thrown whenever your process limits are exceeded (ttl/memory).
  • CODE_INVALID_ENTRY_FILE - thrown when your entry file is missing, or an error prevents load.
  • CODE_INVALID_DEFAULT_FUNCTION - thrown when you don't export a default function, or if your default export is not a function.
  • CODE_INVALID_MIDDLEWARE_FUNCTION - thrown when one or more of your middleware functions are invalid (e.g. not functions).
  • CODE_BUNDLE_ERROR - thrown when bundling errors occur (currently unused).
  • CODE_FATAL_ERROR - thrown when an unhandled exception/rejection is caught (currently unused)

These will be tagged with {type:"user|system|unknown"}.

Usercode​

When there's an error in your application, it'll be wrapped:

{
  "version": "v1",
  "ok": false,
  "result": null,
  "error": {
    "original": {},
    "name": "Error",
    "message": "bad things happen",
    "stack": "..."
  },
  "duration": 820.5204840000006
}
warning

error.original may only be available inside your process. Use middleware to intercept and handle it if needed.

These will be untagged and instead make best effort to normalise and serialise the error thrown.

Unable to resolve dependencies​

If you're seeing "cannot resolve X" for:

  • esbuild
  • @xgsd/engine
  • @xgsd/workers (if using @xgsd/woker-cli)

Then you need to install the missing dependency in the path of your worker:

yarn add @xgsd/engine @xgsd/workers esbuild

You'll need to do this for each worker as global import isn't supported.

This is an intentional constraint as it eliminates trying to determine where something is imported from.

ESM/CommonJS​

If you're having compatibility issues with ESM/CommonJS consider activating the bundler:

worker.json
{
  "bunder": {
    "enabled": true
  }
}

This should elimate most problems (please report issues here).

Bundler​

When bundler.enabled = true in your worker.json file then your entry file will be bundled with esbuild.

The bundler serves two purposes:

  • Increases portability by outputting a consistent format (ESM)
  • Reduces overhead for TypeScript by turning TypeScript -> ESM to avoid expensive JIT compilation.

Having problems?

Bundle not updating/refreshing​

If you've set your cache strategy to always then the cache value will always be returned regardless of how long it's been kept or if anything has changed.

If your strategy is change and your bundle doesn't refresh then:

  1. You may not be using the right extension (needs to be .js/.ts) - change detection is primitive and only supports .js and .ts to avoid hashing everything.

Worker guard​

Anything to do with CODE_WORKER_GUARD means that one or more of your limits were exceeded and the process has been killed.

If this is unexpected, the default limits are:

{
  "limits": {
    "ttl": 5000,
    "memory": 64
  }
}

Override the defaults in your worker.json file.

Serialisation/Deserialisation​

If you're seeing {} in a result or error that most likely means either:

  1. You're returning an instance or other unserialisable type.
  2. The value you're returning is too large for serialisation.

A trade off to using child processes is that information isn't passed in like arguments to a function. Instead data is serialised and sent over IPC.

As a result, values must be serialised or they'll be reduced to {}.

This experimental feature was added in @xgsd/workers@v0.1.0.
Experimental features may change or be removed without notice.
