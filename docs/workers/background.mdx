## TL;DR

1. A **Worker** is an isolated function that runs input to output predictably.
2. Workers will influence the design of xGSD v1.
3. `@xgsd/runtime` and `@xgsd/cli` are now legacy.
4. Most of xGSD < v0.7 can be replaced with middleware.
5. You should migrate to **Workers** if you're using xGSD.

## Background

xGSD was started to find ways of predictably running usercode functions without a cloud setup.

Between v0.5 and v0.7, xGSD stablised executing usercode functions inside processes/containers without needing a full cloud setup (or mini-cloud).

The shape that was defined is:

```typescript
type RunFn<T, R> = (data: T) => Promise<R>;
```

Over time, this simple runtime grew into end-to-end orchestration with everything included.

A few things didn't change though:

- `RunFn` - remained completed untouched.
- `execute()` - has pretty much remained untouched (minus extraction into `@xgsd/engine` and `withTimeout()` introduction)
- `retry()` - same story with a few minor improvements here and there.

When releasing xGSD v0.7, I found a flaw with its design: **the shift toward extensions and project-scoped execution meant that plugins themselves could cause the system to act unpredictably or crash ungracefully**.

There was no clean solution to support extensions at a project-level whilst maintaining a system that is predictable without enforcing per-project isolation.

This would lead to even longer start up times for an environment that is mainly unneeded. The duplication of state between processes would also mean that devices like the RPi Zero 2 would be basically unsupported.

So I defined how `@xgsd/workers` would compare to `@xgsd/runtime` (see [**Full Comparison**](#full-comparison) for more info):

| **Feature**          | **`@xgsd/runtime`**              | **`@xgsd/workers`**                          |
| -------------------- | -------------------------------- | -------------------------------------------- |
| **Local-first**      | Yes                              | Yes                                          |
| **Run anywhere**     | No                               | Yes                                          |
| **Purpose**          | Full orchestration of "projects" | Serverless / isolated execution of functions |
| **Scope**            | Project-level systems            | Single execution                             |
| **Complexity**       | Higher                           | Minimal                                      |
| **Extensibility**    | High (plugins/events)            | Simple middleware system                     |
| **Execution model**  | Stateful workflow runtime        | Stateless worker runtime                     |
| **Fault tolerance**  | High                             | Critical                                     |
| **Primary strength** | Coordination & structure         | Speed & isolation                            |
| **User experience**  | _Almost_ everything built in     | Nothing built in                             |

Here's what _Workers_ will always get right:

- **bullet-proof execution of a worker in a container environment**.

Here's what we've intentionally excluded:

- **Projects** and **Blocks** or any form of orchestration/sequencing.
- **Retry logic** and soft timeouts (not including `ttl`).
- **Concrete transport adapters** - no pub/sub or req/res layers in `@xgsd/workers`.
- **Templating** and any pre-/post-processing of data.
- **Config syntax**
- **Event-based abstractions** - like `EventBus` or event emitters.

I'll benchmark **Workers** using these metrics:

| Metric                       | Acceptable | Goal     | Note                                                   |
| ---------------------------- | ---------- | -------- | ------------------------------------------------------ |
| **End-to-end response time** | `< 100ms`  | `< 50ms` | This is the time it takes from process start to end    |
| **Base memory usage**        | `< 20MB`   | `< 10MB` | Measured when a **Worker** is in an idle/blocked state |

If you're using `@xgsd/worker-cli`, these metrics may be automatically collected.

:::note
Benchmarking will take place on modest hardware and may vary machine-to-machine.

Feel free to share your results if you benchmark **Workers**.
:::

## Use cases

Here's a few ideas for what you can do with **Workers**:

| Use case                                  | Suitable | Note                                                                                           |
| ----------------------------------------- | -------- | ---------------------------------------------------------------------------------------------- |
| Production, mission critical systems      | **No**   | Workers aren't designed to power huge projects - they're hackable sandboxes for your projects. |
| Home serverless lab                       | **Yes**  | designed for this use case                                                                     |
| ETL/data pipelines                        | **Yes**  | input -> function -> output model fits nicely                                                  |
| Automation/glue layer (like Zapier)       | **Yes**  | near-perfect fit                                                                               |
| Sandboxing/safe execution of unknown code | **Yes**  | everything is isolated making unsafe coder safer                                               |
| Prototyping/system design                 | **Yes**  | rapidly prototype ideas                                                                        |

There's not much you can't do, if you're new to Node.js or simply tired of re-inventing the same logic for small projects then give it a go.

## Middleware vs Extensions

One of the biggest differences is the new middleware system.

Instead of worrying about what an **Executor** is and how that aligns with an **Orchestrator** to override behaviour like retries, you define middleware.

The flow of middlewares is predictable and _everything_ runs in the same container, as shown:

![xGSD Workers architecture](/img/middleware.png)

The outer border indiciates the containers physical boundary. Usually this is inside a child process but could equally run inside something like Docker.

This design ensures both middleware and worker failures are handled the same way resulting in less divergence.

When you compare that to plugins, it's the opposite:

![xGSD Workers architecture](/img/plugins.png)

Plugins were conceptually OK but in reality meant to provide any form of resillience a new container would need to be introduced.

### Retry()

Here's a real world example that replicates xGSD retry logic:

```javascript title="worker.js"
import { retry, withTimeout } from "@xgsd/engine";
export const retryMiddleware = async (ctx, next) => {
  // exponential backoff
  const backoff = (attempt) => 1000 * 2 ** attempt;

  ctx.execute = async (fn) => {
    // using @xgsd/engine retry()
    return retry(ctx.data, fn, 3, {
      timeoutWrapper: withTimeout(1000),
      backoff,
      onAttempt: (attempt) => {
        if (attempt.finalAttempt) {
          console.log(
            `[retry] all hope is lost (${attempt.attempt + 1}/${attempt.maxRetries})`,
          );
          return;
        }

        console.log(
          `[retry] currently retrying, next: ${attempt.nextMs.toFixed(2)} ms (${attempt.attempt + 1}/${attempt.maxRetries})`,
        );
      },
    });
  };

  await next();
};
```

You import `@xgsd/engine` and define what `retry()` looks like without endless configuration.

This was possible with plugins but there was no timeout protection and very minimal protection against failures (e.g. silently drops errors).

Plugins were really just reactors as they were unable to share or mutate state.

:::warning
Mutating `ctx` should be done with care.

Otherwise you'll face endless debugging nightmares.
:::

## Challenges/Learnings

I've compiled some of the learnings from xGSD for anyone trying to make something like **Workers**.

I hope it helps.

---

- **Over-abstraction and incorrect abstraction boundaries**
  - Extensions (plugins, hooks, lifecycle systems) introduced unclear execution order and hidden shared state.
  - Many abstractions were introduced before `@xgsd/engine` stabilised, resulting in systems being bolted on.
  - Feature growth outpaced architectural clarity, causing overlapping responsibilities and blurred boundaries.
  - **Key finding:** KISS — Keep It Simple, Stupid.

---

- **Poor performance and execution overhead**
  - Excessive state sharing between main and child processes led to duplicated memory usage and unnecessary coupling.
  - Block lifecycle transitions (start/stop) introduced measurable overhead (~50–100ms per block in some cases).
  - Dependency bloat from unbounded imports and weak justification increased install size and startup cost, especially on constrained hardware.
  - IPC communication complexity with trying to manage main -> child and main -> project -> child communication.
  - **Key finding:** minimise cross-process state and strictly control serialised data.

---

- **Breaking changes and dependency instability**
  - Tight coupling between `@xgsd/engine`, `@xgsd/runtime`, and CLI tooling meant internal changes often forced CLI updates.
  - Version boundaries were unclear during extraction into separate packages.
  - Despite this, the core user code interface introduced in xGSD v0.2 remained largely stable.
  - **Key finding:** the user-facing execution model was stable; most changes were internal orchestration improvements.

---

- **Unclear mental model**
  - The hardest issue to resolve, as it compounded all other problems.
  - Multiple overlapping concepts (runtime, extensions, plugins, CLI behaviour) made system behaviour difficult to reason about.
  - This also slowed refactoring and made documentation inconsistent (or nonexistent).
  - **Key finding:** clarity of mental model is more important than feature completeness. KISS applies, but so does documenting as part of design, not after implementation.
