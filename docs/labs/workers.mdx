## **Quick Overview**

_Workers_ are functions that execute in an isolated environment.

They are intentionally limited and have no concept of **Projects** or **Blocks**.

Instead of a project-level `config.yaml`, you create a worker-level `worker.json`.

| **Feature**          | **`@xgsd/runtime`**        | **`@xgsd/workers`**             |
| -------------------- | -------------------------- | ------------------------------- |
| **Best for**         | Full orchestration systems | Serverless / isolated execution |
| **Scope**            | Project-level systems      | Single execution                |
| **Complexity**       | Higher                     | Minimal                         |
| **Extensibility**    | High (plugins/events)      | Low (intentionally constrained) |
| **Execution model**  | Stateful workflow runtime  | Stateless worker runtime        |
| **Primary strength** | Coordination & structure   | Speed & isolation               |

Example `worker.json`:

```json title="worker.json"
{
  // not complete
  "entry": "index.ts",
  "bundler": {
    "enabled": true,
    "cache": {
      "strategy": "change"
    }
  },
  "http": {
    "enabled": true
  },
  "limits": {
    "ttl": 1000,
    "memory": 100
  },
  "metadata": {}
}
```

Example `index.ts` (or `.js`):

```typescript title="index.ts"
export default async function (data: any) {
  const url =
    data.url ??
    "https://timeapi.io/api/Time/current/zone?timeZone=Europe/London";

  const res = await fetch(url);
  const json = await res.json();

  return json;
}
```

## Goal

**Reduce memory usage and time wasted starting/stopping processes.**

After refactoring a lot of xGSD for v0.7, I realised there's a lot that xGSD is doing that's not truly needed.

Workers will focus on _need to have_ vs _nice to have_.

| Metric                              | Workers     | Main        |
| ----------------------------------- | ----------- | ----------- |
| Start up time                       | _need test_ | _need test_ |
| Memory usage (base) - all processes | < 40MB      | > 90MB      |
| Memory usage (base) - child process | < 10MB      | > 25MB      |

## Schema

Workers are designed around a static schema:

```json title="schema.json"
{
  "version": "v1",
  "ok": true,
  "result": {
    // data
  },
  "error": {
    // error
  },
  "duration": 0
}
```

Example:

```json title="schema.json"
{
  "version": "v1",
  "ok": true,
  "result": {
    "year": 2026,
    "month": 5,
    "day": 9,
    "hour": 5,
    "minute": 51,
```

Wrapping results/errors and serialisation is managed for you. This is what will be returned in API/CLI responses.

## Transports

_Workers_ are transport-agnostic and you can create new transport options using `createHandler()`:

```javascript title="yourapp.ts"
import { createHandler } from "@xgsd/workers";

const config = getWorkerConfig();
const handler = createHandler(config);

// inside your transport (e.g. Express route):
async function callback(req, res) {
  const data = req.body;

  const result = await handler({
    data,
    env: {
      // env vars
    },
    cwd: "path/to/worker",
  });
}
```

This will be simplified as development continues.

## Errors

**Error handling** is a core concern of _Workers_ and is something it'll do _very well_.

### System

System errors are thrown when something goes wrong _before_ or _after_ your code has finished running.

- `CODE_WORKER_GUARD` - this error is thrown whenever your process limits are exceeded (`ttl`/`memory`).
- `CODE_INVALID_ENTRY_FILE` - thrown when your entry file is missing, or an error prevents load.
- `CODE_INVALID_DEFAULT_FUNCTION` - thrown when you don't export a default function, or if your default export is not a function.
- `CODE_INVALID_MIDDLEWARE_FUNCTION` - thrown when one or more of your middleware functions are invalid (e.g. not functions).
- `CODE_BUNDLE_ERROR` - thrown when bundling errors occur (currently unused).
- `CODE_FATAL_ERROR` - thrown when an unhandled exception/rejection is caught (currently unused)

### Usercode

When there's an error in your application, it'll be wrapped:

```json
{
  "version": "v1",
  "ok": false,
  "result": null,
  "error": {
    "original": {},
    "name": "Error",
    "message": "bad things happen",
    "stack": "..."
  },
  "duration": 820.5204840000006
}
```

## Middleware

_Extensions_ aren't supported in _Workers_, instead a lightweight middleware system is provided:

```typescript
// define your middleware
const logger = (ctx: RuntimeContext, next: Next) => {
  console.log("my function just started");

  // optionally mutate before state:
  ctx.input = { myVar: true };

  await next();

  // do something with result:
  ctx.result = { ...ctx.result, addedByMe: true };

  console.log("my function has ended!");
};

// this function is called just before your function runs
// it should always return an array of middleware functions
export const middleware = () => [logger];
```

## Full Comparison

| **Feature**                      | **`@xgsd/runtime`**      | **`@xgsd/workers`**       |
| -------------------------------- | ------------------------ | ------------------------- |
| **Primary model**                | Workflow orchestration   | Isolated worker execution |
| **Execution style**              | Project/runtime driven   | Request/task driven       |
| **Projects**                     | Yes                      | No                        |
| **Config scope**                 | Project-level            | Worker-level              |
| **Plugins/extensions**           | Yes                      | No                        |
| **Event system**                 | Yes                      | No                        |
| **Middleware**                   | No                       | Core abstraction          |
| **Retries/backoff**              | Built-in                 | User-defined if needed    |
| **Hard TTL support**             | Complex/shared lifecycle | Native/simple             |
| **Process isolation**            | Optional/configurable    | Primary model             |
| **In-process mode**              | Yes                      | No                        |
| **Child-process execution**      | Supported                | Core                      |
| **API/server workloads**         | Secondary                | Primary                   |
| **Queue/job workloads**          | Supported                | Natural fit               |
| **Long-running orchestration**   | Yes                      | No                        |
| **Request/response lifecycle**   | Not primary              | Primary                   |
| **Cold start performance**       | Moderate                 | Fast                      |
| **Memory footprint**             | Higher                   | Low                       |
| **Startup complexity**           | Higher                   | Minimal                   |
| **Runtime extensibility**        | High                     | Intentionally minimal     |
| **Event coordination**           | Yes                      | No                        |
| **Execution observability**      | Rich/runtime-wide        | Per-worker                |
| **Failure isolation**            | Mixed depending on mode  | Strong                    |
| **Serverless-style workloads**   | Possible                 | Designed for              |
| **AWS Lambda style usage**       | Awkward                  | Natural                   |
| **Edge/worker deployment model** | Less suitable            | Suitable                  |
| **Default mental model**         | Project/blocks           | Worker/function           |
| **Shared foundation**            | `@xgsd/engine`           | `@xgsd/engine`            |
