:::tip

**Workers.js** runs your code in isolated processes for predictable execution.

This reduces development time and allows you to focus on _building_.

When paired with your own transporter — it effectively gives you a serverless-style environment that runs across your local network.

:::

---

## Install

Install `@xgsd/workers` with npm or Yarn:

```bash
# using npm
npm i @xgsd/workers --save

# using yarn
yarn add @xgsd/workers
```

---

## Quickstart

Create a new folder for your code:

```bash
mkdir my-worker
cd my-worker
```

Create a `worker.js`:

```js title="worker.js"
// a worker is a simple async function
// it must be the default export
export default async (data) => {
  return { hello: data.name };
};
```

To call your worker:

```js title="app.js"
import { createTransport } from "@xgsd/workers";

// you'd usually do this during app bootstrap
const transport = createTransport({
  entry: "./worker.js",
  contractVersion: "v1.1",
});

// this would usually be inside a callback
// like Express/Koa/etc, really whatever you need
// inside a script works perfectly too!
const result = await transport({ name: "world" });
```

Then:

```bash
node app.js
```

## Schema

Workers are designed around a schema:

```json title="schema.json"
{
  "activationId": "act_",
  "version": "v1.1",
  "ok": true,
  "result": {
    // data
  },
  "error": {
    // error
  },
  "duration": 12.34
}
```

Return values and errors will be wrapped and returned in this format.

Set `output.mode` to `raw` to return unwrapped values/errors.

## Signals

When workers run they produce a stream of **Signals**.

**Signals** are logs, errors, activations, system events, and anything else that is produced as a side effect of your worker running.

The format is consistent and `meta` reflects any signal-specific data.

```json
{
  "ctx": "{id}",
  "act": "{act}",
  "pid": "{pid}",
  "type": "log|error|activation|system|generic",
  "message": "{message}",
  "meta": {},
  "timestamp": 1778318464737
}
```

Create your own **Signals** by logging to `stdout` or `stderr`.

Make sure each log is a single line using `JSON.stringify` for objects.

See [**Signals**](/workers/signals) to learn more.

## Middleware

You can add middleware that is executed before and after your worker runs.

This is similar to Express or Koa middleware.

Here's an example:

```js title="worker.js"
export default async () => {
  console.log("worker");
};

const a = async (ctx, next) => {
  console.log("a");
  await next();
  console.log("a");
};

const b = async (ctx, next) => {
  console.log("b");
  await next();
  console.log("b");
};

const middleware = async () => [a, b];
```

If you run this worker — you'll get:

```text
a
b
worker
b
a
```

This is how you take a simple worker function and extend it with loggers, retry logic, validation, and more.

See [**Middleware**](/workers/middleware) for more information.

## Worker example

There's an API running that exposes a worker that converts JSON strings or objects into the opposite form.

You can call this API at `https://workers-test-api.xgsd.io`.

Any request works but this section covers `POST`.

In the body of your request send:

```json
{
  "convert": {
    "my": "object"
  }
}
```

Or:

```json
{
  "convert": "{\"name\":\"my string\"}"
}
```

Output:

```json
{
  "activationId": "act_6f45be686c12",
  "version": "v1.1",
  "ok": true,
  "result": {
    "name": "my string"
  },
  "error": null,
  "duration": 60.14
}
```

## Terms

Here's how we describe **Workers.js**:

| Term                          | Context                                                            |
| ----------------------------- | ------------------------------------------------------------------ |
| **Workers** or **Workers.js** | Refers to the runtime as a whole (packaged as `@xgsd/workers`)     |
| **Worker**                    | This is everything inside the container - your code and middleware |
| **Process** or **Container**  | Bounded runtime environment/sandbox                                |
| **Middleware**                | Think Express/Koa middleware, it's very familiar                   |
| **Signals**                   | Runtime generated feedback - logs, errors, other events            |
| **Transport**                 | External integration layer - http, bullmq, etc                     |
| **Worker guard** or **Guard** | Container policy enforcement/hard limits                           |
